# AGENTS.md — Clinic Software Starter

## Quick Commands

```bash
# Dev (DB must be running first)
docker compose up -d db          # Start PostgreSQL only
npm run start:dev                # API on localhost:3000

# Full Docker dev (API + DB in containers)
docker compose up -d --build

# Verify
npm run build                    # TypeScript compilation
npm run lint                     # ESLint + auto-fix
npm run test                     # Unit tests (Jest)
```

## Architecture

- **NestJS 11** + **TypeORM** + **PostgreSQL 16**
- Multi-tenant: every data table has `clinic_id` FK
- Schema managed via `clinic_schema.sql` (NOT TypeORM sync — `synchronize: false`)
- Entities auto-loaded from `src/database/entities/*{.ts,.js}` via glob in `database.module.ts`

### Module Structure

```
src/
├── auth/          # Login, JWT strategy, guards
├── users/         # CRUD (protected by JwtAuthGuard)
├── common/        # Shared: DynamicResponseMessage, ResponseInterceptor, AllExceptionsFilter
├── database/      # TypeORM config, entities, seeds
└── main.ts        # Bootstrap: ValidationPipe + ResponseInterceptor + AllExceptionsFilter
```

## API Response Format

All endpoints return `DynamicResponseMessage<T>`:

```json
{
  "status": true,
  "apiStatusCode": 200,
  "message": "Operación exitosa",
  "errors": [],
  "data": { ... },
  "timestamp": "2026-07-13T01:00:00.000Z"
}
```

- Errors are caught by `AllExceptionsFilter` and wrapped in the same format
- Validation errors go in `errors[]` array, message becomes `"Error de validación"`

## Auth Flow

- `POST /auth/login` → token returned in `Authorization` header (NOT in body)
- Protected routes use `@UseGuards(JwtAuthGuard)`
- JWT payload: `{ sub, email, clinicId }`

## DTO Convention

```
dto/
├── request/       # What the client sends (e.g. LoginRequestDto)
└── response/      # What the client receives (e.g. UserResponseDto, excludes passwordHash)
```

- Use `class-transformer` `@Expose()` / `@Exclude()` — never return `passwordHash`
- Response DTOs go in `response/` folder, request DTOs in `request/`
- All validation messages in Spanish

## Environment

- `.env` is gitignored — use `.env.example` as template
- Docker Compose uses `${POSTGRES_USER}`, `${POSTGRES_PASSWORD}`, `${POSTGRES_DB}` from `.env`
- `JWT_SECRET` must be set before starting the app (server-side only, never exposed to client)
- Admin seed auto-runs on startup: creates `ADMIN_EMAIL` user with `ADMIN_PASSWORD`

## Anti-Patterns to Avoid

- **Never use `synchronize: true`** — schema is managed by `clinic_schema.sql`
- **Never hardcode DB credentials** in `docker-compose.yml` — always use env vars
- **Never return raw entities** — always transform through Response DTOs with `plainToInstance()`
- **Never put `access_token` in response body** — it goes in the `Authorization` header
- **Never skip `@UseGuards(JwtAuthGuard)`** on protected routes
- **Never use `import { Response }` in decorated positions** — use `import type { Response }` to avoid `isolatedModules` error
- **Never add comments unless asked** — code should be self-documenting

## Entity Conventions

- Entities are in `src/database/entities/` (generated by `typeorm-model-generator` from existing DB)
- `clinic_id` is NOT NULL on all data tables (multi-tenant constraint)
- Audit columns on every table: `created_at`, `created_by`, `updated_at`, `updated_by`, `status`
- UUIDs generated by PostgreSQL (`uuid_generate_v4()`), not TypeORM

## Testing

- Unit tests: `src/**/*.spec.ts` (Jest 30)
- E2E tests: `test/` directory (Jest + Supertest)
- DB must be running for any test that touches repositories
